Shelby the Barista WhatsApp Ordering System PRD
Overview
This document defines a low-cost but production-minded MVP for a WhatsApp-based ordering, FAQ, and order-status system for Shelby the Barista in HSR Layout, Bengaluru. The goal is to reduce peak-hour counter friction, answer repetitive customer questions quickly, and create a clean path from pilot to scalable production without rebuilding the core architecture. WhatsApp Cloud API uses webhooks for inbound and status events, making it suitable for an event-driven backend.[cite:184][cite:185] Razorpay Payment Links can be created and managed via API/webhooks, which makes them a low-friction payment option for a pilot.[cite:189][cite:192] Supabase Realtime can stream Postgres changes to the staff dashboard for live order updates.[cite:190][cite:193]
Product goals
Reduce counter order-taking load during weekend rush hours.
Reduce average time from customer intent to order confirmation.
Provide deterministic order capture over WhatsApp.
Give staff a simple dashboard to manage incoming orders.
Preserve a modular architecture so any provider can be replaced later.
Success metrics
25% reduction in average order-taking time during pilot peak hours.
20% of peak-hour orders initiated through WhatsApp within 4 weeks.
Less than 3% order error rate on WhatsApp-originated orders.
Median time to first response under 10 seconds for automated replies.
At least 70% of FAQ chats resolved without staff intervention.

Core flows
Flow 1: FAQ
Customer opens WhatsApp chat.
Bot presents quick actions: View menu, Order pickup, Check order status, Ask a question.
If FAQ intent is detected, the system retrieves answers from the café knowledge base.
If confidence is low, the system offers staff handoff.
Flow 2: Pickup order
Customer selects Order pickup.
Bot shows categories.
Customer selects item(s), quantity, and allowed modifiers.
Bot shows cart summary and total.
Customer confirms name and phone number.
System creates order and returns order ID, ETA, and optional payment link.
Staff dashboard receives the order in real time.
Staff updates status to Accepted, Preparing, Ready.
Customer receives status updates.
Flow 3: Dine-in assist
Customer scans table QR or starts chat.
Bot asks for table number.
The rest of the flow mirrors pickup ordering.
Order ticket includes dine-in marker and table number.
Flow 4: Order status
Customer selects Check order status.
Bot asks for order ID or fetches recent active order from the phone number.
Bot returns current status and ETA.

System architecture
Logical components
WhatsApp Provider LayerMeta Cloud API directly or a BSP such as Interakt/WATI/Twilio for pilot.[cite:159][cite:167][cite:169]
Webhook GatewayPublic endpoint that validates events and writes raw payloads.
Orchestrator / State MachineDetermines next state based on session context and message intent.
Business ServicesMenu service, cart service, order service, payment service, notification service.
FAQ Retrieval ServiceRetrieval over approved café docs, with optional LLM answer formatting.
Data LayerPostgres (Supabase) for transactional entities and logs.
Realtime Staff UIReact dashboard listening to order/status changes via Supabase Realtime.[cite:190]
Architecture principles
Channel-agnostic internal schema.
Event log first, side effects second.
Deterministic business rules for order creation.
Replaceable providers behind adapter interfaces.

Webhook design
WhatsApp inbound webhook
GET /webhooks/whatsapp for verification token challenge.[cite:184][cite:185]
POST /webhooks/whatsapp for inbound messages and status events.[cite:184]
Processing rules:
Verify signature/token.
Write raw payload to messages_raw or log store.
Normalize into internal event model.
Deduplicate by provider event/message ID.
Push async processing job.
Razorpay webhook
POST /webhooks/razorpay
Verify signature.
Match payment link/payment entity to order.
Update payment_status.
Emit internal event and optionally send customer confirmation.[cite:189][cite:192]

Risks and mitigations
Risk: kitchen is the real bottleneck
Mitigation: measure counter delay separately from kitchen prep delay before claiming success.
Risk: AI gives wrong answers
Mitigation: retrieval-only answers, low-confidence fallback to staff, no AI control over price/order state.
Risk: menu drift
Mitigation: one source of truth for active items; staff toggle availability in dashboard.
Risk: WhatsApp provider lock-in
Mitigation: internal adapter layer and normalized event schema.
Risk: payment confusion
Mitigation: launch with pay-at-counter first if staff prefers operational simplicity.
SOPs for pilot
Staff checks dashboard continuously during pilot windows.
One person owns manual handoff chats.
Availability updated before every rush window.
Every cancelled order gets a reason code.
End-of-day review of failed chats and wrong orders.
Production-readiness checklist
Provider abstraction complete.
Webhook signatures verified.
Idempotency keys in place.
Order event audit log complete.
Dashboard role separation in place.
Monitoring for webhook failures, queue lag, and message send errors.
Incident runbook documented.
Go-live recommendation
Launch with pickup ordering and FAQ first. Dine-in table ordering should follow only after staff consistently handles pickup flow. This keeps operational change small and makes pilot learning clearer.
